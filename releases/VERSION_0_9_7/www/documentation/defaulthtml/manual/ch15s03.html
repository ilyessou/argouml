15.3. Use Case

A use case represents a complete meaningful “chunk” of activity by the system in relation to its external users (actors), human or machine. It represents the primary route through which requirements are captured for the system under construction

Within the UML metamodel, use case is a sub-class of classifier.

The use case icon is an oval (see Figure 15.1).

15.3.1. Use Case Details Tabs

The details tabs that are active for use cases are described in Table 15.4.

Table 15.4. Details tabs for use case

TabMeaning
ToDoItem

Standard tab.

Properties

See Table 15.5 and Table 15.6 below.

Documentation

Standard tab.

Style

Standard tab. Fill color is used for the use case oval.

Source

Standard tab. It would not be usual to provide any code for a use case, since it is primarily a vehicle for capturing requirements about the system under construction, not creating the solution.

Constraints

Standard tab. There are no standard constraints defined for UseCase within the UML metamodel.

Tagged Values

Standard tab. In the UML metamodel, UseCase has the following standard tagged values defined.

  • persistence (from the superclass, Classifier). Values transitory, indicating state is destroyed when an instance is destroyed or persistent, marking state is preserved when an instance is destroyed.

    Tip

    In general the instantiation of use cases is not a major aspect of any design method (they are mostly concerned with requirements capture. For most OOA&D methodologies, this tag can safely be ignored.

  • semantics (from the superclass, Classifier). The value is a specification of the semantics of the use case.

  • derived (from the superclass, ModelElement). Values true, meaning the use case is redundant—it can be formally derived from other elements, or false meaning it cannot.

    Note

    Derived use cases still have their value in analysis to introduce useful names or concepts.

Note

The UML Element metaclass from which all other artifacts are derived includes the tagged element documentation which is handled by the documentation tab under ArgoUML

The fields on the property tab for use cases are described in Table 15.6.

Table 15.6. Property fields for use case

FieldMeaning
Name

Text box. The name of the use case. Since a use case is a classifier, it would be conventional to Capitalize the first letter (and initial letters of any component words), e.g. RemoteSensor.

Note

ArgoUML does not enforce any naming convention for use cases

Stereotype

Drop down selector. Use case is provided by default with the UML standard stereotypes (metaclass, powertype, process, thread, utility) for classifiers. ArgoUML also provides the additional stereotypes, machine, organization and person. Stereotyping can be useful when creating use cases in the problem domain (requirements capture) and solution domain (analysis), but none of the pre-defined stereotypes are well suited to this.

Warning

If a stereotype is selected in ArgoUML it will not be displayed in the diagram.

Navigate Stereotype

icon. If a stereotype has been selected, this will navigate you to the stereotype property panel (see Section 14.4).

Namespace

Text box. Records the namespace for the use case. This is the package hierarchy.

Button 1 click on the entry will take you to the package defining this namespace (or the model for the top level namespace).

Modifiers

Check box, with entries Abstract, Final and Root.

  • abstract is used to declare that this use case cannot be instantiated, but must always be specialized.

    Caution

    Whilst use cases can be specialized and generalized, it is not clear that an abstract use case has any meaning. Perhaps it might be used to indicate an use case that does not itself interact with an actor, but whose children do.

  • final indicates that this use case can have no further children, while root indicates it is a top level use case with no parent.

Generalizations

Text area. Lists use cases which are generalizations of this one. Will be set whenever a generalization is created on the diagram. Button 1 click on a generalization will navigate to that generalization. Button 2 click gives a pop up menu with five options.

  • Open. Where a generalization exists, navigate to it.

  • Add. Add a new generalization. Note that the new association is added to the model, but not the diagram. You will need to find it in the navigation pane and add it from there.

    Tip

    It is far easier to add generalizations by drawing them on the relevant diagram in the editing pane.

    Caution

    It is possible to add multiple generalizations. However is is hard to see a sensible meaning for this. In addition the size of the text area makes this very inconvenient in ArgoUML.

  • Delete. Where a generalization exists linking to this use case, this removes the link to this use case.

    Tip

    This does not remove the generalization from the model or the diagram, leading to potentially very confusing diagrams. It is usually far easier to remove generalizations from the editing pane.

  • Move Up. Only available where there are two or more generalizations listed (see caution above), and the generalization selected is not at the top. It is moved one up the list.

  • Move Down. Only available where there are two or more generalizations listed (see caution above), and the generalization selected is not at the bottom. It is moved one down the list.

Extends

Text box. Lists any class that is extended by this use case. In ArgoUML at present extension on a use case diagram is through stereotyping a dependency relationship, which will not show in this box.

Where an extends relationship has been created (see below), button 1 click will navigate to that relationship. Button 2 gives a pop up menu through which an anonymous extends relationship can be created.

  • Open. Where an extends relationship has been created, navigate to it.

  • Add. Add a new anonymous extends relationship. This is just a documentation artifact, not a true relationship, where you can record details of the extension.

    Caution

    It is possible to add multiple extends relationships. However is is hard to see a sensible meaning for this. In addition the size of the text area, and the lack of move up/move down options on the pop up menu, make all but the top entry inaccessible in ArgoUML.

  • Delete. Where an extends relationship has been created through use of Add pop up menu, this removes it.

Includes

Text box. Lists any class that this use case includes. In ArgoUML at present inclusion on a use case diagram is through stereotyping a dependency relationship, which will not show in this box.

Where an includes relationship has been created (see below), button 1 click will navigate to that relationship. Button 2 gives a pop up menu through which an anonymous includes relationship can be created.

  • Open. Where an includes relationship has been created, navigate to it.

  • Add. Add a new anonymous includes relationship. This is just a documentation artifact, not a true relationship, where you can record details of the extension.

    Caution

    It is possible to add multiple includes relationships. However is is hard to see a sensible meaning for this. In addition the size of the text area, and the lack of move up/move down options on the pop up menu, make all but the top entry inaccessible in ArgoUML.

  • Delete. Where an includes relationship has been created through use of Add pop up menu, this removes it.

Extension Points

Text box. If this use case is, or can be extended, this field is used to document the points at which such extensions can occur.

Where an extension point has been created (see below), button 1 click will navigate to that relationship. Button 2 gives a pop up menu through which an anonymous extension points can be created.

  • Open. Where an extension point has been created, navigate to it.

  • Add. Add a new anonymous extension point. This is a documentation artifact, where you can record details of the extension point.

    Note

    At present there is no way to give a name to an extension point in ArgoUML.

  • Delete. Where an extension point has been created through use of Add pop up menu, this removes it.

Associations

Text area. Lists any associations connected to the use case. Button 1 click on an association will navigate to that association. Button 2 click gives a pop up menu with five options.

  • Open. Navigate to the association.

  • Add. Add a new association. Note that the new association is added to the model, but not the diagram. You will need to find it in the navigation pane and add it from there.

    Tip

    It is far easier to add associations by drawing them on the relevant diagram in the editing pane.

  • Delete. Where an association is linked to a use case, this removes the link to the use case.

    Tip

    This does not remove the association from the model or the diagram, leading to potentially very confusing diagrams. It is usually far easier to remove associations from the editing pane.

  • Move Up. Only available where there are two or more associations listed, and the association selected is not at the top. It is moved one up the list.

  • Move Down. Only available where there are two or more associations listed, and the association selected is not at the bottom. It is moved one down the list.

Operations

Text area. Listing the operations that are available on this use case. As a sub-class of the UML classifier metaclass, a use case has many of the properties of a class (see Section 16.5).

In some OOA&D processes it might be useful to treat use cases as classes as well, and give them operations and attributes. This is of very limited in ArgoUML, since it is not possible to take a use case and add it to a class diagram, so all class relationships must be shown with the notation of a use case diagram.

Should you wish to add operations to a use case, button 1 click allows you to navigate to them and button 2 click gives a pop up menu with options to Open, Add, Delete, Move Up and Move Down (see Generalizations for examples of how these operations are used).

Attributes

Text area. Listing the attributes that are available on this use case. As a sub-class of the UML classifier metaclass, a use case has many of the properties of a class (see Section 16.5).

In some OOA&D processes it might be useful to treat use cases as classes as well, and give them operations and attributes.
