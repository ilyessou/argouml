16.10. Association

An association on a class diagram represents a relationship between classes, or between classes and an interface.

Caution

ArgoUML is restricted to binary associations at present. See Section 16.10.1 for more on this.

Associations are described under use case diagrams (see Section 15.4). There are a few additional issues specific to their use on class diagrams.

Associations are permitted between interfaces and classes, but UML 1.3 specifies they must only be navigable toward the interface—in other words the interface cannot see the class.

Caution

The current release of ArgoUML does not enforce this. In particular if an association is created where the class end is selected before the interface end, ArgoUML will create an association that is navigable in both directions. A critic will trigger to complain about this.

As with use case diagrams, associations with the tag derived have their value in analysis to introduce useful names or concepts, and in design to avoid re-computation.

As with use case diagrams, associations are often not named, since their meaning is obvious from the context.

As with use case diagrams, there is little value in creating more than two association ends, since ArgoUML does not support multi-way associations.

16.10.1. Three-way and Greater Associations

UML 1.3 provides for associations that are governed by a third associative class. ArgoUML provides no support for multi-way associations or associative classes at this stage.
