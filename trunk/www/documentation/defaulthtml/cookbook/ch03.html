This chapter contains a description on how ArgoUML should work and behave for the users.

These things might not be implemented yet and the solutions might not even be clear but it is a definition of the goal.

The fact that it is not implemented or doesn't work as stated here should be registered as a bug in the bug registering tool.

3.1. Requirements for Look and feel

This describes how the ArgoUML look and feel shall behave.

  1. When multiple visual components are showing the same model element they shall be updated in a consistent manner throught the application.

  2. As soon as the model element changes then all views shall update.

  3. For text fields the update of model element shall not take place until the field has looses focus.

  4. Pressing escape in a field that is edited shall restore the value in the text field to that value when the field got focus, the field should then lose focus without updating the model element.

  5. If a text field requires parsing to extract particular values or confirm its validity then it shall appear in red while the field remains invalid.

    As soon as it becomes valid it should revert to the normal field colour.

    A consequence of this is that the field must be parsed and checked to be valid on each keystroke without updating any model element.

  6. When the field loses focus and the text is valid then the model element should be updated.

  7. If the field loses focus while it is invalid then the field should be returned to its original state as if the escape key was pressed. There should be an audible warning and a user friendly error message displayed in red in the status bar followed by the text that the user attempted to parse.

  8. There shall be no indication of an exception on the screen or in the log if it has occured merely because of a user mistyping or not being aware of UML syntax.

  9. All text fields shall have context sensitive help.

    As follows:

    1. A tooltip that explains the data and format expected by the particular field.

      This can be omitted when there is a header stating the data of the field and the format is obvious.

    2. Pressing F1 or choosing help from the menu shall display a popup window explaining for data and format required by the current input field. Input focus shall be left on the field during any user interaction with the popup (dragging, scrolling or closing).
