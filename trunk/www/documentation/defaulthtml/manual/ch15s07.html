15.7. Dependency

Dependency is a relationship between two artifacts showing that one depends on the other.

Note

UML 1.3 does not include dependency as a relationship on a use case diagram. For historic reasons, ArgoUML has implemented extends and includes as a stereotype of the dependency relationship (they are types of relationship in their own right under UML 1.3).

Under ArgoUML extends is represented as a dependency with an extendsstereotype. An extend relationship from A to B indicates the behavior of B may be augmented by the the behavior of A at a particular extension point or points.

Note

ArgoUML has no formal representation of extension points. They are held as anonymous documentation within a dependency.

Note

It is easy to confuse extends relationships between use cases with generalization. However extends is about augmenting a use case's behavior at a specific point. Generalization is about specializing the behavior throughout the use case.

Under ArgoUML includes is represented as a dependency with an includesstereotype. An include relationship from A to B indicates the behavior of A will include the behavior of B at a location defined by A.

Within the UML metamodel, Dependency, Extend and Include are all sub-classes of Relationship.

Dependency is represented as a dotted arrow from the dependent to the depending use case (see Figure 15.1).

15.7.1. Dependency Details Tabs

The details tabs that are active for dependencies are as follows.

Note

There is no source tab, since there is no source code that could be generated for a dependency relationship.

ToDoItem

Standard tab.

Properties

See Section 15.7.2 and Section 15.7.3 below.

Documentation

Standard tab.

Style

Standard tab

Note

The values for the bounds of the dependency appear to have no meaning. Changing them has no effect on the diagram. This is sensible behavior, given the dependency is tied to a particular use case.

Constraints

Standard tab. There are no standard constraints defined for Dependency, Extend or Include within the UML metamodel.

Tagged Values

Standard tab. In the UML metamodel, Extends, Includes and Dependency have no tagged values of their own, but through superclasses have the following standard tagged values defined.

  • derived (from the superclass, ModelElement). Values true, meaning the dependency, includes or extends relationship is redundant—it can be formally derived from other elements, or false meaning it cannot.

    Note

    Derived dependencies, includes and extends relationships still have their value in analysis to introduce useful names or concepts.

Note

The UML Element metaclass from which all other artifacts are derived includes the tagged element documentation which is handled by the documentation tab under ArgoUML

15.7.2. Dependency Property Toolbar

Go up

Navigate up through the package structure of the model. For a dependency this will be the package containing the dependency.

Go back

Navigate back to the previous artifact.

Warning

This feature is currently broken within ArgoUML and there is no guarantee where (if anywhere) this button will take you.

Go forward

Navigate forward to the next artifact.

Warning

This feature is currently broken within ArgoUML and there is no guarantee where (if anywhere) this button will take you.

Delete

This deletes the selected dependency from the model.

Warning

This is a deletion from the model not just the diagram. To delete a dependency from the diagram, but keep it within the model, use the main menu Edit/Remove From Diagram (Ctrl-R).

15.7.3. Property Fields For Dependency

Name

Text box. The name of the dependency.

Tip

It is quite common to leave dependencies unnamed in use case analysis, since they are only being used to carry the extends, and are not represented on the diagram.

Note

ArgoUML does not enforce any naming convention for associations.

Note

There is no representation of the name of a dependency on the diagram.

Stereotype

Drop down selector. Dependency has no standard stereotypes of its own under UML 1.3. ArgoUML provides two stereotypes extend and exclude to allow depend to be used to draw these relationships. ArgoUML also provides the additional stereotypes, machine, organization and person. The stereotype is shown between « and » above or across the generalization.

Tip

Stereotyping of dependencies on use case diagrams is used solely for the purpose of displaying includes and extends relationships.

Navigate Stereotype

icon. If a stereotype has been selected, this will navigate you to the stereotype property panel (see Section 14.4).

Namespace

Text box. Records the namespace for the dependency. This is the package hierarchy.

Button 1 click on the entry will take you to the package defining this namespace (or the model for the top level namespace).

Suppliers

Text area. Lists the “depending” ends of the relationship, i.e. the end supplying what is needed by the other end.

In the case of extends this is the use case being augmented. In the case of includes it is the use case doing the including.

Button 1 click on a supplier will navigate to that use case. Button 2 click gives a pop up menu with three options.

  • Open. Where a supplier exists, navigate to it.

  • Move Up. Only available where there are two or more suppliers, and the supplier selected is not at the top. It is moved one up the list.

  • Move Down. Only available where there are two or more suppliers listed, and the supplier selected is not at the bottom. It is moved one down the list.

Clients

Text area. Lists the “depending” ends of the relationship, i.e. the end supplying what is needed by the other end.

In the case of extends this is the use case providing the augmenting behavior. In the case of includes it is the use case being included.

Button 1 click on a client will navigate to that use case. Button 2 click gives a pop up menu with three options.

  • Open. Where a client exists, navigate to it.

  • Move Up. Only available where there are two or more clients, and the client selected is not at the top. It is moved one up the list.

  • Move Down. Only available where there are two or more clients listed, and the client selected is not at the bottom. It is moved one down the list.
