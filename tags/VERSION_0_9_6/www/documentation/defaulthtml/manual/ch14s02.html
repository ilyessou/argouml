14.2. The Model

The model is the top level artifact within ArgoUML. In the UML meta-model it is a sub-class of package. In many respects within ArgoUML it behaves similarly to a package (see Section 16.2).

Note

ArgoUML is restricted to one model within the tool.

Caution

In principle you can add a model to a class diagram using the context sensitive (button 2) pop up menu Add To Diagramfrom the navigation pane. However this is very demanding of memory and may cause a problem with ArgoUML running out of memory.

Where standard data types, classes and packages are loaded (the default, see Chapter 22. ArgoUML Built In DataTypes, Classes, Intefaces and Stereotypes), these are sub-packages of the model.

Note

In version 0.9.6 of ArgoUML these sub-packages are not initially visible from the model property pane as owned packages. However if you use the data type navigator from a suitable pane, you can navigate up to the model, and thereafter that data type's sub-packages will appear.

14.2.1. Model Details Tabs

The details tabs that are active for the model are described in Table 14.1. Details tabs for the model.

Note

The Style tab is grayed out, since the model typically has no representation on a diagram (see the caution above).

Table 14.1. Details tabs for the model

TabMeaning
ToDoItem

Standard tab.

Properties

See Table 14.2. Property tool bar for the model and Table 14.3. Property fields for the model below.

Documentation

Standard tab.

Source

Standard tab. This contains a template package declaration appropriate to the entire model. In the future this would be part of the code generation activity.

Constraints

Standard tab. There are no standard constraints defined for Model within the UML meta-model.

Tagged Values

Standard tab. In the UML meta-model, Model has the following standard tagged values defined.

  • derived (from the superclass, ModelElement). Values true, meaning the class is redundant—it can be formally derived from other elements, or false meaning it cannot.

    Note

    Derived models still have their value in analysis to introduce useful names or concepts, and in design to avoid re-computation.

Note

The UML Element metaclass from which all other artifacts are derived includes the tagged element documentation which is handled by the documentation tab under ArgoUML

This appears to be broken in the current release of ArgoUML. Even if it did work, it would be of limited value, since the abstraction property sheet has no mechanism for adding supplier and client artifacts.

Delete

This has no effect—it is meaningless to delete the model!

The fields on the property tab for models are described in Table 14.3. Property fields for the model.

Table 14.3. Property fields for the model

FieldMeaning
Name

Text box. The name of the model. The name of a model, like all packages, is by convention all lower case.

Note

The default name supplied to a new model by ArgoUML, untitledModel, is thus erroneous and guarantees that ArgoUML always starts up with at least one problem being reported by the design critics.

Stereotype

Drop down selector. Model is provided by default with the UML standard stereotypes for model (systemModel and metamodel) and package (facade, framework, stub), as well as the stereotypes for models and packages in the UML standard profiles for software development process (use-case model, analysis model, design model, implementation model, use case package and analysis service package). ArgoUML also provides the additional stereotypes, machine, organization and person.

Note

A number of standard stereotypes are currently missing from ArgoUML: topLevel (from package), use case system, analysis system, analysis package (all three from the software development process profile).

Caution

ArgoUML provides the stereotype realize for models and packages. This appears to be an error, since this stereotype properly belongs to the Association metaclass.

Stereotyping models is a useful thing, although it is of limited value in ArgoUML where you have only a single model.

Navigate Stereotype

icon. If a stereotype has been selected, this will navigate you to the stereotype property panel (see Section 14.4).

Namespace

Text box. Records the namespace for the model. This is the package hierarchy. However since the model is at the top of the hierarchy in ArgoUML, this box is always empty.

Modifiers

Check box, with entries Abstract, Final and Root.

  • abstract is used to declare that this model cannot be instantiated, but must always be specialized.

    Tip

    The meaning of abstract applied to a model is not that clear. It might mean that the model contains interfaces or abstract classes without realizations. Since ArgoUML only permits one model, this is not a meaningful box to check.

  • final indicates that this model can have no further subpackages, while root indicates it is the top level model.

    Tip

    Within ArgoUML root only meaningfully applies to the Model, since all packages sit within the model. In the absence of the topLevel stereotype, this could be used to emphasize that the Model is at the top level.

Derived

Text box. Lists any derived class. Meaningless for a model.

Extends

Text box. Lists any use case that is extended by this model. Meaningless for a model.

Implements

Text area. Lists any models for which this model is the realization. Of little value within ArgoUML, since multiple models are not permitted.

It is possible to add realizations from this menu, although since there is no way to add a model to the other end of the abstraction, it is meaningless. Button 1 click allows you to navigate to them and button 2 click gives a pop up menu with options to Open, Add, Delete, Move Up and Move Down (see Generalizations for examples of how these operations are used).

Owned Elements

Text area. A hierarchical listing of the packages, classes, interfaces, datatypes, actors, use cases, associations, generalizations and stereotypes within the model. For each of these types of artifact, where there is at least one example in the model, a folder icon is shown.

Button 1 double click on the folder icon () opens the folder to reveal the artifacts owned under that category.

Button 1 click on any of the revealed artifacts yields navigates to that artifact.

Button 1 double click on an open folder (with artifacts displayed) will close the folder and hide the artifacts.

Alternatively, Button 2 double click on any item gives a pop up menu.

  • Add. This option creates a new artifact of the specified type within the model, and navigates to that item.

    Note

    The item is placed under its relevant folder, irrespective of where the pop up menu is revealed. So button 2 lick on Actor folder and selection of Add/Package will place the new package under the Package folder, not the Actor folder.

    This is the only menu option actually available to the model (the others apply to packages within the model, see Section 16.2).

    Tip

    It is often far easier to add artifacts by drawing them on the relevant diagram in the editing pane.

  • Delete. This is always grayed out for a model.

  • Go. This is always grayed out for a model.

  • Open. This is always grayed out for a model.
