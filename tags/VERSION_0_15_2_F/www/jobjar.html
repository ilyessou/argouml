JobJar: Your contribution to ArgoUml's Success

ArgoUml depends on the contributions of volunteers to grow and maintain itself. This page will explain how to contribute.

This page may get out of date quickly, so check the change control database, issueZilla, in order to find out the latest defects and enhancements to complete.

Prepare yourself:

  • Read about ArgoUml, the user manual and the developer pages
  • Try out ArgoUML and make issues of things that are buggy or awkward.
  • Try to use it for a small design project. What is useful about the tool? What are the requirements?
  • Read about UML, get the UML1.3 specification from O.M.G.
  • Understand Meta-Modelling, read about UML meta-model implementations (NSUML / MDR). ArgoUml currently uses NSUML, but it will migrate to MDR for version 0.16
  • Understand how diagrams are displayed using the GEF library.
  • Understand the status of the ArgoUml project, which components are mature and which are waiting for a re-design. Look at the source code (use cvs), developer mailing list, and IssueZilla issues.
  • Focus on one part of ArgoUml in the beginning. ArgoUml is a large project, specialize in order to understand well one component.
  • Ask questions first then proceed with some concrete work.

ArgoUml is not all about Java or UML development, this project needs a wide range of skills including:

  • Developers (obviously...!)
  • Analysts
  • Administrative helpers

Special Note to Students

Each task could be made into an undergraduate term project or part of a masters degree project. Working on a free, open source software project such as ArgoUML can make your school project much more impressive, satisfying, and clarify potential benefits.

  • Why start with nothing if you could build on the work of many students and professionals?
  • Why end your term project with just a grade when you could make a visible difference?

The point of a term project is to get experience working on a "real" project. Using ArgoUML will give you experience with over 60,000 lines of Java code that implements a real application.

Proposed Contributions:

  • UML Diagrams (, support for) [ IssueZilla / Cookbook ]
    • Implement a graph layout algorithm that takes into account the conventions of UML diagrams. There is already a layout algorithm for class diagrams. see issue 1954, 1898.
    • Implement arc routing that users find flexible and convient. This could be contributed to the GEF project and used in ArgoUml.
    • Implement the UML artifacts that ArgoUML still does not support, see UML support, also look in issueZilla.
    • Implement diagram types that are part of UML but not supported by ArgoUML yet, see UML support, see issue 1377, 1378.
    • Fix diagram export, issue 11, 207.
    • Improve text editing, issue 1195, 373, 1261, 1537
    • It is not possible to have several lines in a UML Note, issue 504.

  • UML Meta-Model [ IssueZilla / Cookbook ]

    (18 Oct 2003) ArgoUml is in the process of migrating to a new metamodel, MDR. There is a lot of work that needs to be done. However, ask on the developer list before attempting any work in this component.

    • Add an JDBC or EJB interface to the UML meta-model (although we still use NSUML, it would be best to add this feature to MDR) so that models can be stored in databases. (Work started on JDBC/NSUML but it was removed from cvs because it was not maintained)

  • Class Diagram [ IssueZilla / Cookbook ]
    • Fix Stereotypes, see issue 1873, 2178
    • Implement the 'lollipop' notation for Interfaces, issue 1401.
    • Fix Class icon presentation, issue 897
    • Improve Copy/ paste support, issues 1861, 1897

  • Forward Engineering [ IssueZilla / Cookbook ]

    Java forward engineering is supplied by default and there are modules for other languages. It is also strongly linked with 'Notation's, where there is a Java and UML notation for parsing strings.

    • Implement code generation for an additional programming language.
    • Implement support for generating Java code that follows JavaBeans conventions.
    • Implement an extensible and customizable library for code generation in many languages.
    • Implement code generation for UML state machine models, work on this has been done in a Masters Thesis.
    • Generate assert() statements to check OCL constraints on a UML model at run-time.
    • Implement a function to publish a complete view of the project in a set of graphically enhanced HTML file. This should include a choice of graphical output i.e. SVG, PNG, etc.

  • Reverse Engineering [ IssueZilla / Cookbook ]

    Java reverse engineering is supplied by default and there are modules for other languages.

    • Review and improve the ANTLR Java grammar.
    • Improve .jar reverse engineering, this is the Classfile module.
    • Improve how types are resolved in the model, issue 1669
    • Implement C++ reverse engineering, possibly using an ANTLR C++ grammar.
    • Implement J2EE reverse engineering, can you reverse engineer An Enterprise Application (.ear file)? Can you reverse engineer a deployment diagram based on configuration files (Apache, App Server)?

  • User Interface [ IssueZilla / Cookbook ]
    • Document the requirements for ArgoUml in terms of use cases and in the cookbook and use manual.
    • Implement table views (instad of a diagram for example) of UML artefacts. These were partly implemented, but they have now been removed until someone can work on it until completion.
    • Implement Undo / redo, issue 1834. Co-ordinate with this for diagrams (GEF project).
    • Make ArgoUML work within Netbeans and Eclipse as a plugin, issue 420, 862.

  • User Documentation [ IssueZilla / Cookbook ]
    • Write additions to the User's manual for ArgoUML, focus on a specific task or audience.
    • Update the important pages linking to the home page.
    • Contribute questions and answers to the Users FAQ.

  • Documenation and Instructions for the developers of ArgoUML [ IssueZilla / Cookbook ]
    • Write additions to the Developer's Cookbook to help new people contribute to the ArgoUML project.
    • Write requirements / specification, e.g. how the yet unimplemented features should work in detail.
    • Improve the source code JavaDoc.
    • Add to the Developer Zones docs.

  • Object Constraint Language [ IssueZilla / Cookbook ]
    • Write an evaluation of the changes required due to the migration to the new Metamodel, MDR.
    • Enhance ArgoUML's OCL evaluator.
    • What features are needed in order to make writing and evaluating constraints easier?

  • Cognitive Support for Design [ IssueZilla / Cookbook ]
    • Implement some critics or checklist items that give advice you think is useful, eg issue 1558.
    • Implement wizards to help resolve critic items, issue 1557 , 2108.
    • Fix critics that fire at the wrong time, issues 667, 1129, 1573, ...
    • Fix the text descriptions of Critics in the ui, issues 915, 1749, ...
    • Refactor and clean up the critics source code implementation.

  • Testing [ Cookbook ]
    • Implement test scripts using JUnit. Focus on a specific class or group of classes.
    • Help VERIFY fixed issues so that they can be closed. Note that you can only verify an issue is fixed once a release has been made with that fix included. This is a very useful contribution to help the developers focus on developement.

  • Performance
    • Measure what are the limits of ArgoUml's performance with large models.
    • Use a Memory profiler, or Tracer to detect inefficient algorithms.

Suggest a New Task!

Contribute a new task idea to the ones listed above by entering it in Issuezilla as a feature request or task.
